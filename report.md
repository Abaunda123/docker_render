Лабораторная работа №12 (Git).

<h1>Цель работы:</h1> <h2> Изучить основы работы с системой контроля версий Git и платформой GitHub, включая создание и инициализацию локального репозитория, фиксацию изменений, управление ветками и выполнение операций их слияния, а также размещение статического веб-сайта в удалённом репозитории на GitHub.    </h2>

<h1>Выводы и документация:</h1>

* <h2>Процесс выполнения:</h2>
Для выполнения лабораторной работы была установлена система контроля версий Git, после чего в выбранной директории был инициализирован новый локальный репозиторий. В рамках репозитория был создан файл report.md, содержащий описание выполнения лабораторной работы. Данный файл был добавлен в систему контроля версий и зафиксирован с помощью коммита. Для дальнейшего редактирования и структурирования отчёта была создана отдельная рабочая ветка. В ней выполнялось дополнение и корректировка содержимого файла отчёта, после чего изменения были сохранены. По завершении работы с отчётом рабочая ветка была объединена с основной веткой проекта, в результате чего актуальная версия документа стала частью главной ветки репозитория.

  После завершения работы с локальным репозиторием был создан удалённый репозиторий на платформе GitHub с названием homework_IIT. Между локальным и удалённым репозиториями была настроена связь, после чего все зафиксированные изменения были отправлены на сервер. Далее в репозиторий были добавлены файлы статического веб-сайта, разработанного ранее, и выполнена их фиксация.

  В завершение была выполнена настройка GitHub Pages, что позволило опубликовать статический сайт.


* <h2>Трудности:</h2>
Основные трудности при выполнении лабораторной работы возникли при попытке отправки проекта в удалённый репозиторий. При выполнении команды push возникали ошибки аутентификации (Authentication failed), связанные с тем, что GitHub больше не принимает пароли при работе через консоль. Для решения данной проблемы на GitHub была настроена авторизация с использованием Personal Access Token, что потребовало изучения принципов генерации токенов, задания прав доступа и корректной их передачи при выполнении операций с удалённым репозиторием.

  Сложности также возникли при публикации сайта с помощью GitHub Pages. Для корректного размещения требовалось выбрать подходящую ветку и указать расположение файлов сайта. Изначально статические файлы находились во вложенной директории репозитория, однако GitHub Pages не распознавал указанный путь (также нужно было включить GitHub Pages в настройках репозитория), поэтому файлы сайта были перенесены в корневой каталог репозитория.

  Кроме того, в процессе работы возникали типовые сложности, связанные с пониманием логики работы веток, последовательности команд и особенностей синхронизации локального и удалённого репозиториев. Для корректного выполнения операций приходилось внимательно отслеживать состояние репозитория и порядок выполнения команд, что потребовало дополнительного времени на практическую отработку действий.


* <h2>Вывод:</h2>
Выполнение данной лабораторной работы позволило подробно познакомиться с принципами работы системы контроля версий Git и платформы GitHub, а также понять значимость организованного управления изменениями даже в рамках небольших проектов. В ходе работы стало очевидно, что процесс разработки выходит за рамки локального хранения файлов: каждая правка фиксируется, каждая версия проекта сохраняется, а история изменений становится прозрачной и контролируемой.

  В процессе выполнения лабораторной работы было установлено, что Git представляет собой не просто средство резервного копирования файлов, а полноценную систему, требующую внимательного отношения к структуре проекта, порядку выполнения команд и логике работы с версиями. Также на практике были изучены основные понятия, такие как коммиты, ветки, слияние изменений и синхронизация репозиториев. 

  Дополнительным практическим результатом стало размещение статического сайта с использованием GitHub Pages, что позволило увидеть полный путь проекта — от исходных файлов до доступного в сети веб-ресурса. Таким образом, лабораторная работа дала целостное представление о инструментах управления версиями и показала их практическую ценность.
